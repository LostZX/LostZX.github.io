浅蓝师傅的payload

膜一下，浅蓝师傅提出了新的链，目光从RCE转到任意文件写入

https://b1ue.cn/archives/382.html

先来看下@浅蓝师傅首先提出来的一条链，实例化三个类就能完成任意文件写入，这里恰恰满足fastjson的调用逻辑

首先是safeFileOutputStream，可以看到创建了OutputStream，这里只要调用到this.output.write和flush就成了

然后浅蓝师傅找到了com.esotericsoftware.kryo.io.Output类，这个类的flush方法同时调用了write和flush，看到构造方法，buffer和position都可以直接设置

最后找到了一个能调用到flush方法的ObjectOutputStream，在构造方法中调用了自身定义的drain方法，然后会调用到out的write方法，这里的out自然就是Output这个类，在Output的write方法
中又调用了flush方法，这也一条链就串起来了。

看一下fastjson的payload，这里的$ref，就不是之前讲的调用getter方法了，而是传递对象的引用，即$.stream实际上引用的是steam中的SafeFileOutputStream对象，这里还有几个点，一个是buffer是一个byte参数，会自动base64解码，所以传入也要base64编码，SerialOutput只有一个两个参数的构造方法，但这里只赋值了out参数，另一个会赋值null

长亭师傅的payload

首先是LocalVariableTable这个东西，咱也不懂，跳过了，直接去看怎么构造的吧，有兴趣的师傅可以参考原文

https://mp.weixin.qq.com/s/6fHJ7s6Xo4GEdEGpKFLOyg

以下分析全部建立在看懂了大师傅的分析流程，然后正向分析一波，我是菜比，没有挖链的能力，只会bb

先来分析一下流程，java写入一个文件，正常就是获取到输入流的内容，然后写入到输出流，然后write，flush，这样子就是获取到入口点，能任意写入的输入流，能输出的输出流，能将输入流转换成输出流的方法。

先看最简单的，将输入流转换成输出流，用的就是org.apache.commons.io.input.TeeInputStream类，这样写入就变成调用TeeInputStream.read()方法。

然后看到入口点，用的是org.apache.commons.io.input.XmlStreamReader，并且接收一个InputSteam的构造方法，很适合作为入口，这里给出它的调用链，最终会调用到一个read方法
XmlStreamReader<init>(InputStream,String,boolean,String) -> doHttpStream() -> bom.getBOMCharsetName() -> getBom() -> this.in.read()(这里的in再bom初始化的时候是BufferedInputStream，所以调用的是BufferedInputStream的read方法) -> fill() -> getInIfOpen().read(byte,int,int) 这样就调用到了InputSteam.read，配合上面的TeeInputStream只用找到一个输入流和输出流，再利用这个类转换就可以完成调用。

然后找InputSteam，org.apache.commons.io.input.ReaderInputStream这个类的构造方法，接收一个Reader参数，执行read方法时会调用fillBuffer方法，会调用到this.reader.read(char[], int, int)，从而获取到输入内容，然后用org.apache.commons.io.input.CharSequenceReade作为Reader，调用read(char[], int, int)时读取所有内容，这样就完成了从自定义字符串里读输入的InputStream。

这样再去找最后一个输出流，找到的是WriterOutputStream，继承了OutputSteam可以做TeeInputSteam的输出流，然后它在执行 write 方法时，会执行 flushOutput 方法，从而执行 Writer.write(char[], int, int)，通过 Writer 来输出。最后找到一个输出到文件的流org.apache.commons.io.output.FileWriterWithEncoding，调用write(char[], int, int)方法就会写入文件

然后就能组装成完整payload，这样执行确实能创建文件，但是写入不了内容，需要手动调用writerOutputStream.flush()去触发他真实的写入方法，实际上fileWriterWithEncoding.write的写入会被保存在缓冲区，这当然不可能完成，需要找到一个方法完成自动flush，看到write的实现，调用的是OutputStreamWriter.write，最终调用到StreamEncoder.impWrite，这里会判断写入的玩意是否占满缓冲区，如果缓冲区沾满了，就会调用真实的write方法写入文件，但是，XmlStreamReader的构造方法中BufferedInputStream的buffer的长度已经固定了4096，这样就只能往OutputStream里写三次，超过他默认的8192，这样就就做到了写入文件。

转成fastjson的方式，多个new XmlStreamReader实际上就是多次引用输入和输出

最后这里有个小细节，这里的json是个畸形json，看了几篇文章，结合自己的理解大概讲一下，fastjson追求效率的时候，牺牲了很多对于json结构的判断，{"@type":"java.lang.String""aaaa"比如这样一个json，fastjson首先会解析到{符号，然后标记当前token为12就是左花括号，接下来会去用默认的解析器解析json，然后拿到"符号进行扫描符号，找到下一个"其中的内容作为key，然后获取到:，这是，fastjson就会进行预测，因为一般@type后面跟的是字符串，就会直接去扫描到下一个字符串，就是需要反序列化的类，这里是String类，就会获取到StringCodec反序列化器，这里为了解析速度，会进行预测下一个字符串是否是"，如果是，直接扫描下一个"获取中间的字符串，并且直接返回，但是如果这里是,就会进入parser，然后匹配字符，这里匹配不到,就会报错
